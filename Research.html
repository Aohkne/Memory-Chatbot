Section 1

Bối cảnh — vấn đề bộ nhớ của LLM

LLM (Large Language Model) về bản chất là stateless — mỗi lần gọi API là một cuộc hội thoại hoàn toàn mới, không có ký ức về những gì đã nói trước đó. Đây là rào cản lớn nhất để xây dựng một trợ lý AI thực sự cá nhân hoá.

Ba giải pháp phổ biến trong industry:

  • Full context stuffing — nhét toàn bộ lịch sử vào context window. Đơn giản nhưng tốn token, bị giới hạn bởi context length, không scale.
  • Vector database retrieval — embed tất cả messages, semantic search khi cần. Bắt được ngữ nghĩa tốt nhưng cần infra phức tạp (embedding model, vector DB).
  • Structured memory extraction — LLM tự trích xuất fact từ hội thoại, lưu dưới dạng structured. Nhẹ hơn, interpretable, nhưng phụ thuộc vào chất lượng extraction.

Dự án này theo hướng thứ 3, kết hợp với keyword-based retrieval — ưu tiên đơn giản, không cần vector DB, dễ debug và inspect.

Section 2

Các bài báo nghiên cứu chính

MemGPT: Towards LLMs as Operating Systems
Packer et al. · UC Berkeley · 2023 · arXiv:2310.08560

MemGPT xem LLM context như RAM và external storage như disk — mô phỏng cách OS quản lý bộ nhớ ảo. Khi context window đầy, LLM tự quyết định evict thông tin nào ra "disk" (external storage) và load thông tin nào về "RAM" (context) tuỳ theo task hiện tại.

Hệ thống có 3 loại storage: In-context (main context = RAM), External (archival storage = HDD), Recall (conversation history = có thể search).

Ứng dụng vào dự án: Kiến trúc 3 tầng lưu trữ (MEMORY.md / topic files / .jsonl transcripts) lấy cảm hứng từ phân cấp này. MEMORY.md = in-context (luôn load), topic files = external (load on-demand), .jsonl = recall (search only).
A Survey on the Memory Mechanism of Large Language Model based Agents
Zhang et al. · 2024 · arXiv:2404.13501

Survey phân loại memory của LLM agent thành 4 loại dựa trên tâm lý học nhận thức:

  • Sensory memory — input tức thì (current token window)
  • Short-term memory — in-context conversation history
  • Long-term memory — external storage, persistent qua session
  • Procedural memory — "cách làm" được mã hoá trong model weights

Survey cũng chỉ ra rằng long-term memory hiệu quả nhất khi kết hợp episodic (lịch sử cụ thể) và semantic (fact tổng quát) — tương ứng với .jsonl transcripts và topic .md files trong dự án này.

Ứng dụng vào dự án: Topic files chứa semantic memory (fact tổng quát: "user dị ứng tôm"). Transcripts .jsonl chứa episodic memory (lịch sử cụ thể từng lượt chat).
Sleep-Consolidated Memory (SCM) for LLM Agents
Nghiên cứu lấy cảm hứng từ neuroscience — memory consolidation during sleep

Trong neuroscience, bộ não người consolidate memory trong lúc ngủ: hippocampus replay các sự kiện trong ngày, neocortex dần dần hấp thụ thông tin quan trọng vào long-term memory. Quá trình này giúp loại bỏ noise, giữ lại pattern quan trọng, và kết nối các memory rời rạc.

Áp dụng cho LLM: thay vì consolidate liên tục (tốn tài nguyên, có thể nhiễu), chạy một "giấc ngủ" định kỳ để xử lý batch các hội thoại gần đây.

Ứng dụng vào dự án: autoDream chính là cơ chế "sleep consolidation" — chạy batch định kỳ (≥24h + ≥5 session), đọc toàn bộ transcript gần đây (hippocampus replay), LLM hợp nhất và prune (neocortex consolidation).
Generative Agents: Interactive Simulacra of Human Behavior
Park et al. · Stanford · 2023 · arXiv:2304.03442

Nghiên cứu xây dựng 25 agent mô phỏng hành vi con người trong một "làng ảo". Mỗi agent có memory stream (chuỗi observation có timestamp), reflection mechanism (định kỳ tổng hợp pattern từ memory), và planning (lập kế hoạch dựa trên memory).

Reflection là cơ chế quan trọng nhất: agent không chỉ nhớ fact thô mà còn suy luận từ các fact để tạo ra insight cao hơn ("Klaus is interested in software engineering" → "Klaus would be a good person to talk to about startups").

Ứng dụng vào dự án: autoDream Phase 3 (Consolidate) mô phỏng reflection — LLM không chỉ copy fact mới mà còn hợp nhất với context cũ để tạo ra hiểu biết sâu hơn. extractor (per-turn) = memory stream. autoDream = reflection.
Cognitive Architecture for Language Agents (CoALA)
Sumers et al. · Princeton · 2023 · arXiv:2309.02427

CoALA đề xuất framework thống nhất cho LLM agent, phân tách rõ: Memory (working / episodic / semantic / procedural), Action space (retrieval / reasoning / grounding / learning), và Decision making (planning).

Đặc biệt, CoALA nhấn mạnh tầm quan trọng của learning — khả năng agent cập nhật memory dựa trên experience, không chỉ retrieve.

Ứng dụng vào dự án: Kiến trúc write paths (extractor / autoDream / manual_write) tương ứng với các learning mechanisms trong CoALA: incremental learning (extractor), batch learning (autoDream), và explicit instruction (manual_write).
Section 3

Quyết định kiến trúc dựa trên research

3.1 — Tại sao chọn ngưỡng ≥24h và ≥5 session cho autoDream?

Ngưỡng này không phải tuỳ tiện — có 2 lý do từ research:

  • ≥24h: Neuroscience cho thấy memory consolidation xảy ra trong chu kỳ ngủ (~24h). Chạy autoDream quá thường xuyên (vài giờ/lần) thì tốn tài nguyên và không có đủ "signal mới" để consolidate. Chạy quá hiếm (vài ngày) thì memory lỗi thời quá lâu.
  • ≥5 session: Một session quá ít fact để LLM có đủ context đưa ra quyết định consolidation tốt. Generative Agents dùng threshold tương tự cho reflection trigger. 5 session ≈ đủ để xuất hiện pattern lặp lại hoặc mâu thuẫn đáng xử lý.

Cả 2 điều kiện phải đồng thời đúng (AND) — tránh autoDream chạy quá thường (nếu chỉ dùng session count) hoặc sau kỳ nghỉ dài không chat (nếu chỉ dùng giờ).

3.2 — Tại sao có cả extractor (per-turn) lẫn autoDream (batch)?

Hai cơ chế bổ sung cho nhau:

Cơ chế Tốc độ Chất lượng Khi nào dùng
extractor Ngay sau mỗi turn (async) Trung bình — chỉ thấy 1 turn, thiếu context Fact quan trọng cần lưu ngay (tên, dị ứng, v.v.)
autoDream Batch định kỳ (≥24h + ≥5 session) Cao — thấy toàn bộ pattern, loại bỏ mâu thuẫn Pattern dần hiện ra qua nhiều session (sở thích, thói quen)

Survey (Zhang et al., 2024) gọi đây là immediate encoding vs. consolidated encoding — cả 2 cần thiết trong hệ thống memory production.

3.3 — Tại sao dùng file-based lock thay vì queue hoặc mutex?

File-based lock (.consolidation.lock) được chọn vì:

  • Không cần dependency ngoài (queue library, Redis, v.v.)
  • Visible — có thể inspect thủ công khi debug
  • Persistent qua process restart (khác với in-memory mutex)
  • Có stale detection: nếu process giữ lock bị crash, lock tự được giải phóng sau LOCK_STALE_MINUTES
Hạn chế đã biết: File-based lock dùng polling mỗi 0.2s — không đảm bảo thứ tự (không FIFO). Nếu extractor và autoDream cùng chờ lock, không có gì đảm bảo ai vào trước. Đây là trade-off chấp nhận được vì cả 2 đều chỉ cần "không ghi đồng thời", không cần thứ tự nghiêm ngặt.

3.4 — Tại sao manual write (/remember) không tự giải conflict?

Thiết kế có chủ ý: khi user nói "tôi hết dị ứng tôm", manual_write append nội dung mới vào topic cũ thay vì xóa. Lý do:

  • Tránh mất dữ liệu do LLM phân loại sai (false positive)
  • Giữ lại lịch sử — autoDream Phase 3 sẽ có đủ context để quyết định loại bỏ fact nào
  • Consistent với nguyên tắc "write is cheap, prune is expensive" trong memory systems

3.5 — Tại sao dùng keyword overlap thay vì vector embeddings?

Đây là quyết định đánh đổi có chủ ý:

Keyword overlap (hiện tại) Vector embeddings (production)
Dependency Không cần gì thêm Embedding model (OpenAI / local)
Synonym "bò" ≠ "beef" → MISS "bò" ≈ "beef" → HIT
Debug Dễ — inspect trực tiếp từng từ Khó — cosine similarity là black box
Latency ~0ms (pure Python) ~50–200ms (API call hoặc local inference)
Phù hợp khi Prototype, debug, nhỏ Production, user base lớn, đa ngôn ngữ

Research (Dense Passage Retrieval, Karpukhin et al. 2020) chứng minh semantic search vượt trội BM25/keyword-based cho open-domain QA. Tuy nhiên với personal memory chatbot quy mô nhỏ, keyword overlap đủ dùng và dễ maintain hơn nhiều.

Section 4

So sánh diagram gốc vs. implementation

Trong quá trình phát triển, phát hiện 6 điểm khác biệt giữa diagram thiết kế ban đầu và code thực tế. Research được dùng để xác định implementation nào đúng hơn về mặt lý thuyết.

01 extractor có ghi vào .jsonl không? Code đúng

Diagram gốc: Không có mũi tên từ extractMemories → .jsonl.

Code thực tế: extractor.py không ghi vào .jsonl — chỉ ghi vào topic files và MEMORY.md.

Kết luận: Diagram và code đều nhất quán — extractor không ghi .jsonl. Transcript được ghi bởi main.py (transcript logger), hoàn toàn độc lập với extractor.

02 autoDream đọc hay ghi topic files? Code đúng

Diagram gốc: Chỉ vẽ mũi tên autoDream → topic files (ghi).

Code thực tế: autoDream vừa đọc (Phase 1: Orient) vừa ghi (Phase 3: Consolidate, Phase 4: Prune).

Research backing: Generative Agents (Park et al.) — reflection phải đọc memory trước khi có thể tổng hợp. Đọc mà không có → không thể biết context cũ → dễ tạo duplicate.

03 extractMemories và extractor.py là 1 hay 2 thứ? Cùng nhau

Diagram gốc: Dùng tên "extractMemories".

Code thực tế: extractor.py với function maybe_extract_async() → spawn thread → _run().

Kết luận: Cùng 1 thứ — diagram dùng tên khái niệm, code dùng tên file. extractMemories = extractor._run().

04 Background consolidation dùng "subagent" hay threading? Code đúng

Diagram gốc: Gọi autoDream là "background process" — mơ hồ, có thể hiểu là subprocess hoặc agent riêng.

Code thực tế: threading.Thread — chạy trong cùng process, không phải subprocess hay Claude subagent.

Research backing: MemGPT dùng async executor trong cùng process. Dùng subprocess cho task CPU-light như file I/O + LLM API call là overkill và phức tạp hơn không cần thiết.

05 Race condition khi ghi đồng thời vào cùng file Code đúng

Vấn đề: extractor và autoDream đều ghi topic files. Nếu cả 2 đọc file vào RAM cùng lúc trước khi ai kịp ghi → một bên sẽ ghi đè mất dữ liệu của bên kia.

Code thực tế: .consolidation.lock — file-based lock với stale detection. Cả extractor lẫn autoDream đều phải acquire lock trước khi read-merge-write.

Diagram gốc: Không đề cập race condition — đây là improvement so với diagram.

06 Retrieval dùng mechanism gì? Code đúng (với hạn chế đã biết)

Diagram gốc: Không mô tả rõ retrieval mechanism.

Code thực tế: Keyword overlap — so token giữa query và (hook + title) của từng topic, lấy những topic có overlap ≥ 1 từ (không tính stopwords), sort by overlap descending.

Research backing: BM25 / TF-IDF là baseline mạnh cho keyword retrieval. Với personal memory có số topic nhỏ (<100), keyword overlap đủ hiệu quả. Điểm yếu: synonym miss ("bò" ≠ "beef") — fix bằng vector embeddings khi scale.
